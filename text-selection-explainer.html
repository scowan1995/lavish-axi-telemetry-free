Lavish Editor text selection annotation

Text annotations now use range anchors, not just element selectors.

The editor still supports clicking an element, but now it also detects a highlighted text selection and sends a structured range target back to the agent.

How it works

1
The artifact SDK listens for text selection. On mouseup, it calls document.getSelection() and ignores collapsed or empty selections.
2
The selected text becomes a first-class target. The SDK records the selected text, the common ancestor selector, and start/end DOM boundaries.
3
Boundaries are robust across nested markup. Each boundary stores a selector for the closest element, a child-node path from that element to the exact text node, and the character offset.
4
The same annotation card is reused. The SDK draws a translucent highlight over every client rect in the range, opens the card near the selection, and queues the prompt with tag: "text".
5
The session store preserves the target. Polling now returns the structured target object so the agent can understand exactly which words the user meant.

What if there is no ID?

IDs are helpful but not required. When an element has an ID, Lavish uses it in the selector. When there is no ID, Lavish falls back to a structural selector such as body > main > div:nth-of-type(2).

The selector alone is not the full identity. A text annotation is a locator packet: structural selector, selected text, common ancestor selector, start boundary, end boundary, child-node paths, offsets, and the DOM snapshot returned with the poll. That combination lets the agent find the intended passage precisely in the current artifact, even when the selected phrase crosses nested elements.

The DOM snapshot UIDs are still useful, but they are runtime cross-references, not source-code IDs. They let the prompt point back into the snapshot from the same poll response. The agent should not treat them as durable identifiers after the artifact reloads or after the HTML is edited.
